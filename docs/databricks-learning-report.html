Databricks 学習成果報告

テーマ: Git User Analytics（GitHubコミット履歴のユーザー単位可視化）

小さな題材を通して、Databricksの売りである「取り込みから可視化までを一つの基盤で回せる感覚」を体験することを狙った。

Databricks SQL Notebook Delta Lake Jobs
1. 取得
GitHub API からコミット情報を取得
Notebook 上で Python をそのまま使い、データ取得から始められる。
2. 整形
分析しやすい形にデータを整形
SQL と Python を同じ基盤で組み合わせて処理できる。
3. 集計
日別・時間帯別・ディレクトリ別に集計
生データから集計済みデータまで段階的に管理しやすい。
4. 可視化 / 運用
ダッシュボード化し、Jobで日次実行
可視化と定期実行まで同じ基盤の中でつなげられる。

1. 目的

Databricksの機能を列挙して理解するのではなく、実際に小さなシステムを作りながら「何がDatabricksらしいのか」を体感することを目的とした。

2. まずDatabricksの売りは何か

  • データの取り込み、整形、集計、可視化を同じ基盤で扱える。
  • SQLだけでなくNotebook（Python）を同じ流れの中で使える。
  • Bronze / Silver / Goldのように、分析用のレイヤ設計をそのまま実装に落とし込みやすい。
  • 最初は手動実行でも、後からJobで自動化しやすい。

3. 他のツールで同じことをやるなら

やりたいこと 一般的な構成のイメージ Databricksで今回やったこと
GitHubからデータ取得 別のPythonスクリプトやバッチを用意する Notebook内のPythonでそのまま取得
データ保管 DBやDWHを別途用意する Deltaテーブルにそのまま保存
整形・集計 ETLツールやSQL基盤を別でつなぐ Notebook / SQLで同じ場所で処理
可視化 BIツールに接続してグラフ化する Databricks Dashboardでそのまま表示
自動化 スケジューラやワークフロー基盤を追加する Databricks Jobにまとめられる

つまりDatabricksだけでしかできない、というより「分かれていた役割を一つの基盤に寄せやすい」のが売りだと理解した。

4. AWSでやるならどうなるか

処理 AWSでよくある構成例 今回のDatabricks
GitHub API取得 Lambda か ECS でバッチ実行 NotebookのPythonで実行
生データ保存 S3 Bronze Delta Table
整形・集計 Glue / EMR / Athena Notebook + SQL
テーブル定義やデータ管理 Glue Data Catalog Databricks上でそのまま管理
可視化 QuickSight Databricks Dashboard
定期実行 EventBridge + Step Functions など Databricks Job

AWSでも実現できるが、役割ごとにサービスを組み合わせることが多い。今回Databricksを触ってみて、これらを一つの基盤に寄せて考えられる点が分かりやすいと感じた。

5. 作成したシステム

GitHub API取得
Bronze: raw JSON保存
Silver: 正規化・重複排除
Gold: 可視化向け集計
Dashboard表示
Jobで日次実行

Medallion（Bronze/Silver/Gold）で構成し、Notebook実行によりパイプラインを構築した。さらに Databricks Job による日次実行まで設定した。

6. 実装内容

レイヤ テーブル 役割
Bronze bronze_git_raw GitHubコミット詳細JSONの生データ保持
Silver silver_user_commits / silver_user_changed_files 分析用の列形式へ整形、重複排除（MERGE）
Gold gold_user_activity_daily / hourly / top_directories_30d ダッシュボード表示に最適化した集計テーブル

7. 今回触ったDatabricks機能

機能 今回やったこと 学べたこと
Notebook PythonでGitHub APIを呼び、ETLを3段に分けて実装 SQLだけでなくコード実行も同じ基盤で扱える
Delta Table Bronze / Silver / Gold の各テーブルを作成 生データ、中間データ、集計済みデータを段階的に管理できる
Databricks SQL テーブル確認、集計結果の確認、ダッシュボード用データの確認 処理後のデータ確認と分析を同じ環境で進められる
Dashboard 日別コミット数、時間帯別、ディレクトリTOP10を可視化 整形済みテーブルをそのまま可視化につなげられる
Secrets GitHub PAT を Secret として登録して利用 ノートブックに認証情報を直書きせずに使える
Jobs `ingest -> transform -> aggregate` を日次実行に設定 手動で作った処理をそのまま定期運用に移せる

8. 可視化結果

  • 日別コミット数（Daily Commits）
  • 時間帯別コミット数（Commits by Hour）
  • 30日間の変更ディレクトリTOP10（Top Directories 30d）

上記3グラフをDatabricks Dashboardで作成し、ユーザー単位で活動傾向を確認できる状態まで到達。

9. Job化して分かったこと

  • ETLパイプラインと実行スケジュールを同じ製品の中で閉じられるので、AWSで組む場合に比べると、スケジューラやワークフローを別サービスで考えなくてよい点が分かりやすかった。

10. Databricksを触ってみてどうだったか

  • 第一印象は「BIツールに近い」だったが、実際には可視化よりも前段のデータ処理まで一体で持っているところが印象的だった。
  • GitHub APIの取得、データ保存、SQLによる集計、ダッシュボード作成、Job化までを一連で試せたことで、Databricks上でデータ活用の流れを一通り理解できた。
  • 一方で、「UIのどこで何を設定するか」が分かりづらく、使い慣れるまでには一定の学習コストを感じた。

11. このシステムを作って得た学び

観点 学び
設計 Databricksには、Bronze / Silver / Gold のようにデータを段階的に分けて扱う Medallion Architecture という考え方があることを理解した。
実装 Notebook + SQLでETLと分析を同時に進められ、試行錯誤のサイクルが速い。
運用 Notebookで作った処理を、そのままJobで日次実行に載せられる。
比較理解 Databricksは「BIツール」単体ではなく、データ基盤の上流から下流までをまとめて扱いやすくする製品だと理解できた。

12. 現在の状態

データ取り込み・整形・集計・可視化に加え、Databricks Jobによる日次実行まで設定完了。

現在は `ingest -> transform -> aggregate` を Job で日次実行できる状態であり、ダッシュボードの継続更新が可能。
